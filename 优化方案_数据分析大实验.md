## 数据分析系统综合实践 — 优化方案

> 基于指导书要求 + 16GB Apple Silicon Mac 硬件条件，在原建议基础上修正偏差、补强薄弱环节。

---

### 一、总体评估：原建议哪些对、哪些要改

**保留不动的好设计：**
FastAPI 后端 + Swift/SwiftUI 前端的分离架构；Ollama 托管 DeepSeek 量化模型；Neo4j 图数据库 + 向量数据库的双存储思路；分阶段推进的开发节奏；MVP 优先的心态。

**必须修正的偏差：**

1. **评分重心抓偏** — 指导书写的是"算法和模型难度、精度、系统性能"，原建议重 UI 轻算法。
2. **RAG 双知识库没讲清楚** — 图谱（结构化）和向量库（非结构化）是两条独立的检索通路，需要融合，这是核心难点。
3. **数字人方案与指导书指定技术不匹配** — 指导书指定 Stable Diffusion，不是 ComfyUI 工作流或 SadTalker。
4. **内存估算过于乐观** — 没有给出各组件分时加载的具体策略。
5. **缺少评估方案** — 没有说明如何验证 RAG 准确性和系统性能。

### 三、16GB 内存分时调度策略（核心）

这是整个项目能不能跑起来的关键。16GB 统一内存的分配大致如下：

| 组件 | 常驻/按需 | 内存占用估算 |
|------|----------|------------|
| macOS 系统 + 基础服务 | 常驻 | ~4 GB |
| FastAPI + ChromaDB + Neo4j | 常驻（服务期间） | ~1.5 GB |
| Ollama (DeepSeek 7B Q4) | 按需（对话时加载） | ~5 GB |
| SD 1.5 推理 (diffusers + MPS) | 按需（生成图片时加载） | ~3.5 GB |

**调度原则：Ollama 和 SD 绝不同时驻留内存。**

具体实现方式：
- FastAPI 后端维护一个"模型管理器"，用引用计数或简单的状态机控制模型加载。
- 用户发起对话 → 加载 Ollama 模型 → 生成回答 → 如果回答需要生成数字人视频/图片，先卸载 Ollama，再加载 SD。
- 利用 Ollama 自身的 `OLLAMA_KEEP_ALIVE` 参数控制模型在内存中的驻留时间（设为 0 可以立即释放）。
- SD 推理完成后立即 `del model; torch.mps.empty_cache()` 释放显存。

**如果内存还是不够的降级方案：**
- 将 DeepSeek 从 7B Q4 降到 7B Q3（精度损失可接受，省约 1.5GB）。
- 将 ChromaDB 换成 FAISS（内存占用更小，但功能更基础）。
- 数字人只生成静态图片，不做动态视频。

### 四、各模块详细方案

#### 4.1 知识图谱建设（Neo4j）

**选哪门课？** 建议选择《机器学习》，原因是：概念之间的关系更清晰（前置依赖、算法分类、模型对比），三元组更容易构建，图谱质量更高，老师评审时一看就懂。数据挖掘的概念边界更模糊，构建难度大但展示效果不一定更好。

**本体设计（Ontology）：**

实体类型（节点标签）：
- `Concept` — 核心概念（如"支持向量机"、"梯度下降"）
- `Algorithm` — 具体算法（如"SGD"、"Adam"）
- `Model` — 模型（如"ResNet"、"随机森林"）
- `Metric` — 评估指标（如"准确率"、"F1-Score"）
- `Dataset` — 经典数据集（如"Iris"、"MNIST"）

关系类型（边标签）：
- `PREREQUISITE_OF` — 前置知识（如"线性代数 → 主成分分析"）
- `PART_OF` — 包含关系（如"决策树 → 集成学习"）
- `COMPARE_WITH` — 对比关系（如"SVM vs 逻辑回归"）
- `USED_IN` — 应用场景（如"交叉验证 → 模型选择"）
- `EVALUATED_BY` — 评估关系（如"分类器 → F1-Score"）

**数据来源：** 以周志华《机器学习》（西瓜书）或李航《统计学习方法》的目录结构为骨架，再补充课程 PPT 和笔记中的细节。预计节点 80-150 个，关系 150-300 条。

**查询优化（指导书明确要求）：**
- 为高频查询路径建立索引：`CREATE INDEX FOR (c:Concept) ON (c.name)`
- 编写常用 Cypher 查询：最短学习路径（`shortestPath`）、某概念的完整前置知识链、相似算法的对比图谱。
- 在 FastAPI 中封装这些查询为 RESTful 接口。

#### 4.2 RAG 系统（核心，评分重点）

这是整个项目技术含量最高的部分，必须做扎实。

**双通道检索架构：**

```
用户提问
   │
   ├─→ [通道A: 向量检索]
   │    将问题 Embedding → 在 ChromaDB 中检索 Top-K 相关文本切片
   │    （来源：教材章节文本、PPT内容、课程笔记等）
   │
   ├─→ [通道B: 图谱检索]
   │    从问题中提取关键概念 → 在 Neo4j 中查询该概念的关联知识
   │    （前置知识、相关算法、应用场景等结构化信息）
   │
   └─→ [融合与重排]
        将两路结果合并，用 Reciprocal Rank Fusion 或简单的加权拼接
        → 组装 Prompt → DeepSeek 生成回答
```

**实现要点：**

1. **文档预处理：** 将教材、PPT 等按章节/段落切片（chunk size 建议 300-500 tokens，overlap 50-100 tokens）。用 `text_splitter` 工具（LangChain 的 RecursiveCharacterTextSplitter 即可）。
2. **Embedding 模型：** 用 Ollama 的 `nomic-embed-text` 或 `bge-m3`（约 300MB，可以常驻内存），不要和大语言模型抢内存。
3. **图谱检索的关键——实体识别：** 用一个轻量 NER 或直接用 DeepSeek 做一步 "从问题中提取关键概念"，再用这些概念去 Neo4j 查询。这比 Text2Cypher 更稳定（Text2Cypher 让小模型生成 Cypher 语句容易出错）。
4. **Prompt 工程：** 将两个通道的检索结果分段放入 system prompt，明确告诉模型"基于以下参考资料回答，如果资料不足请如实说明"。

**评估方案（必须有，评分看"精度"）：**
- 手工准备 20-30 个标准问答对（覆盖不同知识点），作为测试集。
- 评估指标：回答的准确性（人工打分 1-5 分）、检索的召回率（是否找到了相关段落）、忠实度（回答是否基于检索内容而非模型幻觉）。
- 做一组消融实验对比：只用向量检索 vs 只用图谱检索 vs 双通道融合，用数据证明双通道的优势。这是报告里最有说服力的内容。

#### 4.3 数字人（Stable Diffusion）

**指导书原话：**"使用免费开源的 Stable Diffusion 模型实现完整的数字人开发、部署、调试"。

**正确理解：** 核心是用 SD 生成数字人形象，不是搭建一个完整的实时 3D 数字人。

**推荐方案（由简到繁，逐步加码）：**

**Level 1（必做，保底）：**
- 使用 HuggingFace `diffusers` 库加载 SD 1.5 模型。
- 设计一组详细的 Prompt，生成一个固定的"虚拟教师"形象（比如 "a professional young female teacher, portrait, warm lighting, clean background, high quality"）。
- 用 ControlNet 或 img2img 基于同一角色生成不同表情/姿态的变体（微笑、讲解、思考等 5-8 张）。
- 前端根据对话内容/情感切换显示不同的图片。

**Level 2（加分项）：**
- 接入 TTS（推荐 `edge-tts`，免费、中文效果好、无需本地 GPU），将 DeepSeek 的回答转为语音。
- 前端在播放语音时同步切换数字人表情图片（伪动态效果）。

**Level 3（锦上添花，量力而行）：**
- 使用轻量级唇形同步模型（如 Wav2Lip 的 Mac 优化版）驱动数字人嘴部。
- 这一步在 16GB Mac 上比较吃力，建议只在答辩演示时作为"技术展示"环节单独运行，不要和 RAG 对话同时跑。

**为什么不推荐 ComfyUI：** ComfyUI 是一个基于节点的 SD 工作流 GUI 工具，它本身不是模型。在你的项目里需要的是代码级别的 SD 调用（通过 API 或 Python 库），而不是在 GUI 里拖拽节点。用 `diffusers` 库可以在 FastAPI 后端直接调用，集成更自然。

#### 4.4 前端（SwiftUI 或 Web）

**原建议的 SwiftUI 方案没问题，但有一个更实际的选择值得考虑：Web 前端。**

理由：
- 知识图谱可视化在 Web 端有成熟的库（vis.js、D3.js、ECharts 的关系图），SwiftUI 下需要自己用 Canvas 画，工作量大且效果不一定好。
- 对话界面的流式输出（SSE/WebSocket）在 Web 端实现更简单。
- 答辩演示时浏览器全屏效果和原生 App 差别不大。

**如果坚持用 Swift（体现原生开发能力是加分项）：**
- 知识图谱可视化可以用 `SpriteKit` 或者嵌入 `WKWebView` 加载 vis.js。
- 对话界面用 SwiftUI 的 `List` + `ScrollViewReader` 实现流式效果。

**如果选择 Web 前端：**
- 推荐 React + Tailwind CSS，或者直接一个 HTML 单页面。
- 三栏布局：左侧知识图谱导航（可折叠）、中间对话区、右侧数字人展示。

### 七、避坑清单

1. **不要在答辩时同时跑所有模型。** 演示对话时只加载 Ollama，演示数字人生成时单独跑 SD。提前录好一段完整的演示视频作为备份。
2. **Neo4j 的 Cypher 语句别用 LLM 生成（Text2Cypher）。** 7B 模型生成的 Cypher 错误率很高，不如自己预定义好查询模板，用参数化查询填充。
3. **SD 1.5 在 MPS 上有已知的 NaN 问题。** 如果遇到生成全黑图，设置 `torch_dtype=torch.float32`（不要用 float16）。
4. **向量库的 chunk 大小直接影响检索质量。** 太大检索不精确，太小丢失上下文。建议从 300 tokens 开始，用测试集调优。
5. **报告里的消融实验数据是加分利器。** 哪怕只做了 20 个 QA 的对比测试，用表格和图表展示出来，远比只说"效果不错"有说服力。
6. **监控内存用 `memory_pressure` 命令** 比活动监视器更方便脚本化：`memory_pressure` 可以实时查看系统内存压力等级。
