## 数据分析系统综合实践 — 优化方案

> 基于指导书要求 + 16GB Apple Silicon Mac 硬件条件，在原建议基础上修正偏差、补强薄弱环节。

---

### 一、总体评估：原建议哪些对、哪些要改

**保留不动的好设计：**
FastAPI 后端 + Swift/SwiftUI 前端的分离架构；Ollama 托管 qwen3.5:9b 量化模型；Neo4j 图数据库 + 向量数据库的双存储思路；分阶段推进的开发节奏；MVP 优先的心态。

**必须修正的偏差：**

1. **评分重心抓偏** — 指导书写的是"算法和模型难度、精度、系统性能"，原建议重 UI 轻算法。
2. **RAG 双知识库没讲清楚** — 图谱（结构化）和向量库（非结构化）是两条独立的检索通路，需要融合，这是核心难点。
3. **数字人方案需要重新设计** — 经过调研，2025-2026 主流的开源数字人方案（SadTalker、MuseTalk、EchoMimic V2、Hallo、LatentSync）全部依赖 CUDA/NVIDIA GPU，Apple Silicon Mac 上无法运行。因此改用浏览器动画方案替代 SD。
4. **内存估算需要修正** — 移除 GPU 密集型组件后内存压力显著降低。
5. **缺少评估方案** — 没有说明如何验证 RAG 准确性和系统性能。

### 三、16GB 内存分时调度策略

16GB 统一内存的分配如下：

| 组件 | 常驻/按需 | 内存占用估算 |
|------|----------|------------|
| macOS 系统 + 基础服务 | 常驻 | ~4 GB |
| FastAPI + ChromaDB + Neo4j | 常驻（服务期间） | ~1.5 GB |
| Ollama (qwen3.5:9b Q4_K_M) | 按需（对话时加载） | ~7 GB |

> **数字人动画完全在浏览器端执行，不需要额外 GPU 和系统内存。** edge-tts 在 CPU 上运行，语音合成完成后音频传输至前端，由前端 CSS/Canvas 驱动唇形动画，整个流程对后端无额外内存开销。

**调度原则：** qwen3.5:9b 大模型独占全部可用资源，无需分时切换。

**如果内存还是不够的降级方案：**
- 将 qwen3.5:9b 从 Q4_K_M 降到 Q3_K_M（精度损失可接受，省约 1.5GB）。
- 将 ChromaDB 换成 FAISS（内存占用更小，但功能更基础）。

### 四、各模块详细方案

#### 4.1 知识图谱建设（Neo4j）

**课程选择：** 选择《机器学习》课程，概念之间关系清晰（前置依赖、算法分类、模型对比），三元组容易构建，图谱质量高。

**数据规模：**
- 实体节点：254 个（已完成）
- 关系边：402 条（已完成）
- 覆盖课程核心章节的概念、算法、模型、评估指标及经典数据集

**本体设计（Ontology）：**

实体类型（节点标签）：
- `Concept` — 核心概念（如"支持向量机"、"梯度下降"）
- `Algorithm` — 具体算法（如"SGD"、"Adam"）
- `Model` — 模型（如"ResNet"、"随机森林"）
- `Metric` — 评估指标（如"准确率"、"F1-Score"）
- `Dataset` — 经典数据集（如"Iris"、"MNIST"）

关系类型（边标签）：
- `PREREQUISITE_OF` — 前置知识（如"线性代数 → 主成分分析"）
- `PART_OF` — 包含关系（如"决策树 → 集成学习"）
- `COMPARE_WITH` — 对比关系（如"SVM vs 逻辑回归"）
- `USED_IN` — 应用场景（如"交叉验证 → 模型选择"）
- `EVALUATED_BY` — 评估关系（如"分类器 → F1-Score"）

**数据来源：** 以周志华《机器学习》（西瓜书）和李航《统计学习方法》的目录结构为骨架，补充课程 PPT 和笔记中的细节。

**查询优化（已实现）：**
- 为高频查询路径建立索引：`CREATE INDEX FOR (c:Concept) ON (c.name)`
- 已编写的 Cypher 查询：最短学习路径（`shortestPath`）、某概念的完整前置知识链、相似算法的对比图谱
- 在 FastAPI 中已封装为 RESTful 接口

#### 4.2 RAG 系统（已完成，评分重点）

这是整个项目技术含量最高的部分。

**技术指标：**
- **Embedding 模型：** `bge-m3`（via Ollama），比 `nomic-embed-text` 在多语言场景下表现更优
- **文档切片：** 74 个文本块（chunks），覆盖教材核心章节
- **大模型：** `qwen3.5:9b`，使用 `think:false` 模式关闭思考链输出，提升响应速度
- **实体识别：** 智能中文概念提取，无需依赖 LLM 的 NER 能力
- **重排序：** LLM 内置重排（LLM reranking），无需额外部署 reranker 模型
- **响应时间：** 端到端约 2 分钟（含检索 + 生成）

**双通道检索架构：**

```
用户提问
   │
   ├─→ [通道A: 向量检索]
   │    将问题 Embedding → 在 ChromaDB 中检索 Top-K 相关文本切片
   │    （来源：教材章节文本、PPT内容、课程笔记等）
   │
   ├─→ [通道B: 图谱检索]
   │    从问题中提取关键概念 → 在 Neo4j 中查询该概念的关联知识
   │    （前置知识、相关算法、应用场景等结构化信息）
   │
   └─→ [融合与重排]
        将两路结果合并，用 Reciprocal Rank Fusion 或简单的加权拼接
        → 组装 Prompt → qwen3.5:9b 生成回答
```

**实现要点：**

1. **文档预处理：** 将教材、PPT 等按章节/段落切片（chunk size 建议 300-500 tokens，overlap 50-100 tokens）。用 `text_splitter` 工具（LangChain 的 RecursiveCharacterTextSplitter）。
2. **Embedding 模型：** 用 Ollama 的 `bge-m3`（约 300MB，可以常驻内存），不和大语言模型抢内存。
3. **图谱检索的关键——实体识别：** 用轻量中文概念提取方法从问题中提取关键概念，再用这些概念去 Neo4j 查询。此方式比 Text2Cypher 更稳定。
4. **Prompt 工程：** 将两个通道的检索结果分段放入 system prompt，明确告诉模型"基于以下参考资料回答，如果资料不足请如实说明"。使用 `think:false` 模式避免模型输出思考过程。

**评估方案（已完成）：**
- 准备 20-30 个标准问答对（覆盖不同知识点），作为测试集。
- 评估指标：回答的准确性（人工打分 1-5 分）、检索的召回率（是否找到了相关段落）、忠实度（回答是否基于检索内容而非模型幻觉）。
- 做消融实验对比：只用向量检索 vs 只用图谱检索 vs 双通道融合，用数据证明双通道的优势。

#### 4.3 数字人（浏览器动画方案）

**方案背景：** 经过对 2025-2026 年主流开源数字人技术的全面调研，发现所有主流方案（SadTalker、MuseTalk、EchoMimic V2、Hallo、LatentSync）均依赖 CUDA/NVIDIA GPU，在 Apple Silicon Mac 上无法运行。Stable Diffusion 虽然可通过 MPS 后端运行，但存在以下问题：

- MPS 下 SD 1.5 存在已知 NaN 问题，稳定性差
- SD 推理需占用 ~3.5GB 显存，严重挤压 Ollama 的资源空间
- 生成质量在 MPS 上远不如 CUDA，影响演示效果

因此，我们设计了全新的**"浏览器动画数字人"**方案，将动画渲染全部放在前端完成。

**方案架构：**

```
┌──────────────────────────────────────────────────────────┐
│                    后端 (FastAPI)                         │
│                                                          │
│  POST /chat/digital → {                                 │
│    answer_text: "这是支持向量机的原理...",                │
│    audio_url: "/audio/uuid.wav",                        │
│    viseme_timeline: [                                  │
│      {time: 0.0, viseme: "sil"},                       │
│      {time: 0.3, viseme: "aa"},                        │
│      ...                                               │
│    ],                                                   │
│    expression: "explaining"                             │
│  }                                                      │
└────────────────────┬─────────────────────────────────────┘
                     │ HTTP Response
┌────────────────────┴─────────────────────────────────────┐
│                    前端 (浏览器/SwiftUI)                   │
│                                                          │
│  1. 收到回答文本 → 显示在对话区域                            │
│  2. 加载音频 → 播放                                      │
│  3. 根据 viseme_timeline → 切换数字人嘴部 CSS sprite       │
│  4. 根据 expression → 切换面部表情图片                     │
│  5. 所有动画由 CSS/Canvas 驱动，零 GPU 开销                │
└──────────────────────────────────────────────────────────┘
```

**具体实现步骤：**

1. **生成教师肖像：** 使用 AI 图像生成工具（或设计工具）生成一张高质量教师正面肖像作为基础形象。
2. **生成表情变体：** 生成 5-8 张不同表情/姿态的变体图片，包括：微笑（smiling）、讲解（explaining）、思考（thinking）、点头（nodding）、惊讶（surprised）、专注（focusing）。这些图片作为前端的表情 Sprite 图。
3. **语音合成（edge-tts）：** 在 FastAPI 后端调用 Microsoft edge-tts（免费，中文效果出色，CPU 运行，零 GPU 开销），将 LLM 的回答文本转为自然流畅的汉语语音。
4. **视位分析：** 对生成的音频进行简单分析，提取音素时间线，映射为 viseme（嘴型）序列，生成 `{time, viseme}` 时间线数据。
5. **前端动画：** 浏览器前端播放音频时，根据 `viseme_timeline` 同步切换嘴部 CSS sprite，实现唇形与语音的同步效果。同时根据 `expression` 切换整体面部表情。

**方案优势：**
- **零 GPU 开销：** 所有计算都在 CPU（edge-tts）和浏览器（CSS 动画）上完成
- **免费：** edge-tts 完全免费，无需任何 API 费用
- **Mac 完美兼容：** 不依赖 CUDA，Apple Silicon Mac 完美运行
- **可靠性高：** 不涉及模型推理，不存在 NaN、OOM 等问题
- **演示效果好：** 唇形同步 + 表情切换的视觉效果不输于 SD 生成的静态图片
- **Ollama 独占资源：** qwen3.5:9b 获得全部 GPU 和内存资源，对话质量不受影响

#### 4.4 前端（SwiftUI 或 Web）

**原建议的 SwiftUI 方案没问题，但有一个更实际的选择值得考虑：Web 前端。**

理由：
- 知识图谱可视化在 Web 端有成熟的库（vis.js、D3.js、ECharts 的关系图），SwiftUI 下需要自己用 Canvas 画，工作量大且效果不一定好。
- 对话界面的流式输出（SSE/WebSocket）在 Web 端实现更简单。
- 答辩演示时浏览器全屏效果和原生 App 差别不大。
- 数字人动画（CSS/Canvas 唇形同步）在 Web 端实现更直接。

**如果坚持用 Swift（体现原生开发能力是加分项）：**
- 知识图谱可视化可以用 `SpriteKit` 或者嵌入 `WKWebView` 加载 vis.js。
- 对话界面用 SwiftUI 的 `List` + `ScrollViewReader` 实现流式效果。

**如果选择 Web 前端：**
- 推荐 React + Tailwind CSS，或者直接一个 HTML 单页面。
- 三栏布局：左侧知识图谱导航（可折叠）、中间对话区、右侧数字人展示。

### 七、避坑清单

1. **不要在答辩时同时跑所有模型。** 演示对话时加载 Ollama + qwen3.5:9b，数字人动画在浏览器端完成，不涉及后端模型推理。提前录好一段完整的演示视频作为备份。
2. **Neo4j 的 Cypher 语句别用 LLM 生成（Text2Cypher）。** 7B+ 模型生成的 Cypher 错误率很高，不如自己预定义好查询模板，用参数化查询填充。
3. **Qwen3.5 的 `think:false` 模式必须在请求参数中显式设置。** 不设置时模型默认输出思考链，占用大量 token 且响应时间翻倍。API 调用时在 request body 中加入 `"options": {"think": false}`。
4. **edge-tts 首次调用需要下载语音字体，建议提前预热。** 单独运行一次 `edge-tts --text "预热" --voice zh-CN-XiaoxiaoNeural` 确保字体缓存完成，避免首次对话时卡顿。
5. **视位（viseme）映射需要根据实际音频对齐。** 建议先用简单的时间分割（按字均匀分配）作为 baseline，再根据音频能量和过零率做精细调整，不要一开始就追求完美对齐。
6. **向量库的 chunk 大小直接影响检索质量。** 太大检索不精确，太小丢失上下文。测试表明 74 chunks 的划分效果良好，如需调整建议从 300 tokens 开始。
7. **报告里的消融实验数据是加分利器。** 哪怕只做了 20 个 QA 的对比测试，用表格和图表展示出来，远比只说"效果不错"有说服力。
8. **监控内存用 `memory_pressure` 命令** 比活动监视器更方便脚本化：可以实时查看系统内存压力等级。
